余弦相似度 v.s. 欧式距离
余弦相似度只看方向不看长度
欧氏距离是点到点的直线距离，若向量已单位化则欧氏距离与余弦相似度等价。（某些训练方法把置信度或强度编码在模长上）
大多数文本/语义 embedding 把语义编码在方向上，归一化后检索效果稳定且易于比较。

HNSW：Hierarchical Navigable Small World 分层可导航小世界图 （分层图）
原理
所有向量组织成多层图结构，上层稀疏允许长距离跳跃，下层稠密确保精度。
插入时随机分配最高层级，然后插入到该层及以下所有层
新向量插入时，找到M个最邻近点，建立边。调整这些最邻近点的边，使各点所连边数不超过M
查询时从入口点开始，在当前层找到局部最优，然后下降到下一层继续搜索，直到最底层。
pros：查询性能极佳；动态插入方便
cons：需要存储大量边，索引大小通常是原始数据的3~5倍；构建速度慢；删除节点会破坏图结构，通常采用逻辑删除，但也会堆积垃圾数据
IVF：Inverted File Index  （ 聚类 + 倒排 ）
原理：
聚类划分：先用K-Means把向量聚类成簇 Cluster，每个簇中心称为Centroid。
倒排映射：建立 Centroid 到向量的映射
查询：计算query与所有Centroid距离，选最近的topk个簇，在簇内暴搜
pros
可与PQ结合，压缩数据
构建快，新数据只要分配到对应的簇
内存可控，可通过调整簇数量和topk来控制内存和精度
cons
精度上限低
延迟比HNSW高
PQ：Product Quantization  （向量压缩编码）
高维向量切成若干子块，每块用聚类中心编号代替原始值。128维向量（512bytes）-> 8段，每段1byte编码 -> 压缩64倍
原理
将D维向量划分成M份子空间，每份长度变为D/M。对每个子空间独立进行K-means聚类，映射到对应的聚类中心（256个D/M维向量），再用码本中聚类中心的index代替，于是原来的D维向量被转换为M个整数。查询时用查表法累加距离（向量 -> 聚类中心编号 -> 查表得到距离）。
码本：离线，一次训练永久使用，与query无关。从知识库中随机采样向量，切成不同段（子空间），对子空间做K-means聚类，得到256个聚类中心，即为该子空间的码本。（4个码本 × 256个聚类中心 × 32维向量 × 4Byte）（向量 -> 聚类中心编号）
查找表：每次query时临时生成。每个子空间内用query和聚类中心计算得到距离。得到的256维距离数组即为该子空间的查找表。（4距离表 × 256个距离值 × 1个float32 ）
若原始是 float32 (4字节)， D=128 ，则需 512字节。若 M=8 ，则只需 8字节（压缩64倍）

LSH：Locality Sensitive Hashing 
设计一组哈希函数，让相似的向量大概率落入同一个桶。查询时只搜目标桶和相邻桶
适用于极高维稀疏向量（如传统推荐系统的用户行为特征）
基本被HNSW全面碾压，现在基本不用了







RAG
检索增强生成 (RAG - Retrieval-Augmented Generation)
LLM的两个问题：
知识截至：LLM的训练数据是有局限性和滞后性的，模型不具备未训练到的知识和最新的知识，进对其具有推理能力。
幻觉：LLM本质是一个条件概率模型，token量较大时可能会生成看似逻辑严谨（概率高）但不符合事实的回答

文档解析
文档清洗
文档分块
自然结构切分：按文档原有格式拆分。
固定大小分块：按固定字符/词/token数切分
滑动窗口：固定窗口+overlap
递归分块：先按大结构粗分，过长的块再用小结构细分
语义切分：保证每个块语义完整独立（用模型判断句子之间的语义相似度，相似度突然下降的地方就是分界点）
混合分块
为什么要分块：
上下文长度有限
提升检索的速度和相关性：一个500 tokens的块比5000 tokens 的块在相似度计算时快很多；比起将整个文档向量化成一个大向量，切成小块后每个块都有各自的向量表示，更容易匹配。
元数据标注
光切块还不够，还要带上：
标题、URL、作者、时间、章节标题、领域标签等

Prompt Engineering
核心目的：让模型基于检索内容回答，减少幻觉
角色设定：“你是一个xx领域的专业助手，只能基于检索内容回答”
结构化模板：检索内容、用户问题、输出格式
兜底：“ 如果资料中找不到答案，回复"根据现有资料无法回答该问题" ”
Few-shot：给几个标准问答样例
自查询 Self-Querying
用户提问往往带着隐含条件，如果直接用户提问去检索，很可能检索出语义相关但关键元数据（时间、作者等）完全不符的文档。
自查询的前提是文档入库时就打好了元数据标签。没有的话就会退回到纯语义检索。
核心思想是利用结构化元数据过滤
查询扩展 Query-Expansion
用户提问可能与知识库中词汇不匹配，或是查询太模糊
流程：用户输入query -> query expansion -> 并行向量检索
核心思想是生成多个语义相似的query / 查询词 提高召回率
实现方式
基于规则的扩展：维护一份同义词词典
调用LLM生成相似query
PRF 基于检索反馈：先用原始query的embedding检索一轮，从topk结果中提取高频词作为扩展词，构造新query

RAG v.s. Fine-tuning
i.e. -- RAG和模型微调有什么区别?
RAG本质上是改变模型输入的上下文；而Fine-tuning则是利用下游知识来进一步训练模型，迭代模型的参数。
RAG更加灵活，知识库可以随时增删改；Fine-tuning一旦完成，知识就固化了。企业级应用一般会选择通过微调来影响模型的行为模式，使其具备特定领域内的通用能力；而使用RAG可以实现具体知识的快速更新。
问答响应速度上RAG更慢，因为每次执行检索都会带来延迟，同时更长的上下文也会让模型处理时间变长。

LangChain 记忆模块
ConversationBufferMemory:最简单的方案，存储完整对话历史，适合对话轮次不多的场景。缺点是历史长了就超窗口
ConversationSummaryBufferMemory:结合短期缓存与长期摘要，最近几轮保留原文，更早的压缩成摘要。在记忆容量和信息保真之间取得平衡
VectorStoreRetrieverMemory:把每轮对话嵌入向量数据库，需要时通过语义检索召回相关片段。适合需要从海量历史中精准定位信息的场景。
ConversationKGMemory:构建知识图谱，抽取实体和关系存储。适合需要追踪多个实体状态的复杂对话，比如客服场景要记住用户的订单号、地址、投诉记录等

CoT 思维链
Chain of Thought
引导模型显式生成中间推理步骤，将复杂问题分解为逻辑连贯的子步骤，从而提升多步推理任务的准确率和可解释性
实现方式：
Zero-shot ：加一句引导语。强模型的零样本CoT越来越好
Few-shot CoT ：prompt里给几个带推理过程的例子
System Prompt：在系统提示词里明确要求展示推理过程
局限：
token 消耗
小模型不适用，中间步骤混乱，反而干扰最终答案
并非所有任务都有效，只适合需要多步推理的问题
推理步骤可能是答案的反向构造。有研究表明模型会在猜到最终答案之后反向编造看似合理的过程。
ReAct 范式
每一步先推理要做什么、为什么这样做，然后再执行，拿到结果再规划下一步
Thought-Action-Observation 循环

工程实现要点
工具描述要精准

最大迭代次数：不设上限的话可能陷入死循环
错误处理：工具调用失败时将错误信息也放入Observation，让模型反思
