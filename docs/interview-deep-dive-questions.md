# Agent/LLM 应用项目 — 面试官深挖问题清单

> 面向「检索-阅读-分析-写作」全流程学术调研系统，区分「跑通 demo」与「真正做明白」的追问。每个问题含：问题原文、浅层回答（易被认为没做明白）、深挖要点、与项目模块关联。

## 一、RAG 维度

### 1.1 切片策略

**问题**：你的 RAG 里，检索内容是怎么切片的？按字符数、按段落、按语义，还是按文档结构？为什么选这种？

- **浅层回答**：「用 LangChain 的 TextSplitter 按 chunk_size 切的。」
- **深挖要点**：能说清（1）**切片的粒度**：字符/句/段/节，以及 chunk_size、chunk_overlap 的取值依据（如 500/1000、overlap 100/200）；（2）**是否按结构切**：如 Markdown 按标题/列表切（MarkdownTextSplitter）、代码按函数/类切，避免把半句话或一个表格拆成两段；（3）**不同数据源的差异**：用户上传文档 vs 论文摘要/提取结果，是否用同一套切片策略；（4）**切片与 embedding 的关系**：一个 chunk 对应一条向量，过小则噪声多、过大则检索粗，你的取舍理由。
- **与项目相关**：**阅读 + 写作强相关**。阅读节点把 `ExtractedPaperData` 的 JSON 整条作为 document、原始 paper 作 metadata 存入；用户知识库用 `kb_utils.split_text_into_chunks`（MarkdownTextSplitter，chunk_size 1000、overlap 200）。需能解释「论文侧为何不按 chunk 切、而按篇存」以及「用户文档侧为何用 Markdown 切」。

---

**问题**：如果一篇长论文或一份 50 页 PDF 要进 RAG，你会怎么切？会不会先摘要再切片，或者先切片再对每个 chunk 做摘要？

- **浅层回答**：「就按长度切呗。」
- **深挖要点**：长文档的典型策略：（1）**先结构后长度**：按章节/小节切，再对超长块二次切；（2）**摘要再切**：先对每节做摘要，用摘要做检索、用原文做生成，减少 token 占用；（3）**分层检索**：先按标题/摘要召回章节，再在章节内做 chunk 级检索。能说明在你当前场景（论文 abstract + 提取字段 vs 全文）下为何没做或做了哪一层。
- **与项目相关**：**阅读节点**：当前是「摘要/提取字段」进向量库，不是全文；若面试官问「为何不用全文」，要能答：篇幅、噪声、成本与召回精度的权衡。

---

### 1.2 检索策略与召回/精准权衡

**问题**：检索时 top_k 和 similarity_threshold 你是怎么定的？召回多了会怎样、少了会怎样？有没有做过调参或 A/B？

- **浅层回答**：「top_k=5，阈值 0.05，感觉够用。」
- **深挖要点**：（1）**top_k**：过大导致上下文冗长、无关片段多、易幻觉；过小漏掉关键文档；（2）**similarity_threshold**：过高压制召回、漏检；过低放进大量低相关 chunk；（3）是否区分**临时库（本次论文）和用户知识库**（如你项目里 tmpdb_top_k=2、tmpdb_similarity_threshold=0.05 vs top_k=5、similarity_threshold=0.05）及理由；（4）是否有**测试集/人工抽检**调参，还是经验值。
- **与项目相关**：**写作/RAG 强相关**。`retrieval_tool` 里临时库与用户库用不同 top_k（config：tmpdb_top_k、top_k）、同一 similarity_threshold；chroma 实现里按 distance 转 similarity 再过滤。需能解释「临时库为何用更小的 top_k」。

---

**问题**：有没有做重排序（rerank）？如果没做，为什么？如果做了，在召回多少条之后 rerank、保留多少条进上下文？

- **浅层回答**：「没做，向量检索够用了。」
- **深挖要点**：向量检索是「粗排」，rerank 是「精排」。能说明：（1）在什么数据规模/查询复杂度下你选择不做（成本、延迟、收益）；（2）若做了，pipeline 是「先 top_k 扩大（如 20）→ rerank → 取 top_n（如 5）」，以及模型/API 选型；（3）项目里是否有 reranker 相关代码或配置（如注释掉的 OnlineReranker），以及为何未启用。
- **与项目相关**：**写作/检索**。`model_client.py` 有注释掉的重排序相关代码，可说明「当前阶段未启用，若加会放在 retrieval_tool 的 aquery 之后」。

---

**问题**：在什么场景下你的 RAG 效果会明显变差？你做过哪些改进或打算怎么改？

- **浅层回答**：「有时候检索不准，换好点的模型会好。」
- **深挖要点**：典型失效场景：（1）**查询与文档表述不一致**（同义不同词），改进：查询扩展、多 query、HyDE；（2）**长文档中细粒度事实**被大 chunk 稀释，改进：更细切片或分层检索；（3）**多跳推理**需要多段组合，改进：多轮检索或图检索；（4）**时效性**：你存的是静态快照，改进：刷新策略或外部 API。结合项目说明「论文调研」场景下最可能失效的是哪类，以及已做/计划做的改进。
- **与项目相关**：**检索 + 阅读 + 写作**。例如：用户问非常细的方法名/指标，而提取字段里是概括性描述，召回可能不足；可提「细化提取 schema 或增加关键词字段」等。

## 二、上下文与 Token 维度

**问题**：上下文超了怎么办？具体是：什么时候发现超、怎么压、压成什么样？

- **浅层回答**：「用 32k 的模型，一般不会超。」
- **深挖要点**：（1）**发现时机**：请求前估算 input token、还是请求后靠 API 报错再重试；（2）**压的手段**：截断（保留 system + 最近 N 轮）、摘要（把旧对话或长文档摘要成一段）、分层（重要信息保留原文，次要用摘要）、丢弃中间节点只保留关键状态；（3）**压成什么样**：是否保证「必须保留的上下文」清单（如当前任务描述、最近工具结果、当前章节要求），再裁其他；（4）是否在代码里做过 token 计数或 max_tokens 配置。
- **与项目相关**：**编排 + 写作 + 分析**。当前各节点把「当步所需」放进 prompt（如 global_analysis、sections），未做显式上下文窗口管理；若分析/写作输入特别大（例如 50 篇论文的聚类结果），需能说明「若超了会怎么处理」的预案（如摘要、分块写再合并）。

---

**问题**：有没有做「长文本分块」或「摘要后再喂给模型」的策略？比如单次请求只喂摘要，需要时再按需拉取原文。

- **浅层回答**：「没有，都是整段喂的。」
- **深挖要点**：能说清（1）**当前**：分析节点/写作节点收到的「论文/分析结果」是完整 JSON 还是已压缩；（2）若存在「先摘要再进 prompt」的环节，摘要的粒度和格式；（3）若没有，在「论文数很多（如 30+）」时的风险（超长、中间信息被忽略）及可落地的改进（按聚类先汇总再给全局分析、写作时按章节只给相关聚类）。
- **与项目相关**：**分析 + 写作**。全局分析收到的是各聚类的 deep_analyse 摘要；写作时 task_prompt 含 user_request + section + global_analyse，若 global_analyse 很长，可讨论「按章节只注入相关聚类」等。

## 三、工具调用维度

**问题**：工具调用用的什么格式（JSON、XML、自然语言）？为什么选它？和别的格式对比过吗？

- **浅层回答**：「用的 AutoGen 默认的，应该是 JSON。」
- **深挖要点**：（1）**实际格式**：OpenAI function calling / AutoGen FunctionTool 一般是 JSON schema；（2）**选型理由**：结构化、易解析、模型支持好；（3）若对比过：与 XML/自然语言相比，JSON 的解析稳定性、错误率、token 开销；（4）工具参数是**模型生成**还是**固定/从上下文填充**（如你的 retrieval_tool 的 querys 是模型填还是从 state 取）。
- **与项目相关**：**写作/检索**。retrieval_agent 使用 FunctionTool(retrieval_tool)，参数为 `querys: List[str]`，由模型生成；需能说明「工具 schema 如何暴露给模型、模型如何构造 querys」。

---

**问题**：工具选错、参数传错、或返回不符合预期时，在「模型看到的上下文」和「节点/状态」层面是怎么表现和处理的？

- **浅层回答**：「会报错吧，没特别处理。」
- **深挖要点**：（1）**选错工具**：例如该查库却调了别的接口，是否在 prompt 里约束「只能调用指定工具」、是否有 fallback；（2）**参数错误**：如 querys 不是列表、或为空，工具层是否校验并返回明确错误信息，该错误是否作为一条 message 塞回对话，模型是否会重试；（3）**返回不符合预期**：如检索为空、或返回格式异常，是否在工具内返回固定说明（如「未找到相关文档」），避免模型误读；（4）**状态层面**：工具失败是否写入 state.error 或仅留在对话里，是否影响下一节点。
- **与项目相关**：**写作/检索**。retrieval_tool 在异常时返回 `{"message": "...", "status": "failed"}`，与正常返回的 `List[List[Dict]]` 不一致，模型可能困惑；可说明「若严格工程化会统一返回结构 + 错误码，并在 prompt 里约定模型对 failed 的处理」。

## 四、Agent Loop / 状态与编排维度

**问题**：你这套流程里，每个节点的「输入上下文」和「输出上下文」分别是什么？跑完第一轮和第二轮后，上下文具体怎么变化？

- **浅层回答**：「就是 state 啊，每个节点读 state 再写回去。」
- **深挖要点**：（1）**LangGraph 层**：每个节点的输入是完整 State（含 state_queue、value），输出是 State 的**部分更新**（如 `{"value": current_state}`），图引擎做合并；（2）**单节点内部**：例如 search_node 的「输入」是 user_request，「输出」是 search_results 写入 state；reading_node 的输入是 search_results，输出是 extracted_data + 写入 KB；（3）**多轮/循环**：若存在循环（如写作组内 writing_agent ↔ retrieval_agent ↔ review_agent），要能说清每轮对话里新增了哪些 message（用户/助手/工具结果），以及何时终止（如 "APPROVE"）；（4）**子图**：写作子图有独立的 WritingState，其「上下文」是 sections、writted_sections、global_analysis 等，与主图 state["value"] 的桥接点在哪（writing_node 内读入/写回）。
- **与项目相关**：**编排 + 写作**。主图无循环，线性 search→reading→analyse→writing→report；写作子图内是 SelectorGroupChat 多轮对话，直到 review 输出 APPROVE。要能画清「主图 state 各节点输入输出」和「写作组内一轮的 message 流」。

---

**问题**：如果工具调用出错（比如检索超时、返回 500），在「上下文」层面会怎样？会不会影响下一轮模型决策？你有没有做重试或降级？

- **浅层回答**：「会报错，可能影响下一轮吧。」
- **深挖要点**：（1）**上下文表现**：工具错误通常作为 tool result 或 error message 插入对话，模型下一轮会「看到」这次失败；（2）**对决策的影响**：模型可能重试、换 query、或放弃检索直接写，取决于 prompt 和实现；（3）**重试**：在工具内部还是 Agent 层做重试（如 tenacity），重试几次、退避策略；（4）**降级**：例如检索失败时是否返回空列表 + 提示「请基于已有内容写作」，避免整链挂掉。
- **与项目相关**：**写作 + 检索 + 阅读**。reading 与 parallel_writing 对 RateLimitError 做了 tenacity 重试（指数退避、最多 5 次）；retrieval_tool 内部异常时返回 failed 结构，但未在工具层重试；需能说清「检索失败时写作组会怎样」（依赖 AutoGen 把 tool 结果塞回对话，模型可继续写或再试）。

---

**问题**：主流程里有没有「循环」？如果有，循环的终止条件是什么？如果没有，为什么设计成线性 DAG？

- **浅层回答**：「没有循环，就是一条线跑完。」
- **深挖要点**：（1）主图是**线性 DAG**：search→reading→analyse→writing→report，条件边只做「下一节点 or 错误节点」，无环路；（2）**子图/子流程**内的循环：写作组内是多轮对话直到 APPROVE，可视为「单节点内的 Agent loop」；（3）若设计成「可重试某节点」（如 reading 失败重跑）：当前是否支持、若支持是通过重新 ainvoke 还是图上有边回到 reading；（4）能说明「为何主流程不做循环」：业务上一步到位、避免无限重试与成本，错误通过 error 状态统一进 handle_error_node。
- **与项目相关**：**编排**。orchestrator 的 condition_handler 是严格顺序 + 错误分支，无 back-edge；要能解释这样设计的原因（可维护性、可观测、Human-in-the-loop 只在 search 一步）。

## 五、错误与鲁棒性维度

**问题**：Agent 的错误（参数错、工具错、结果不合预期、幻觉）和后端运行时错误（网络、超时、500）在你看来有什么不同？你在设计里是怎么区分处理的？

- **浅层回答**：「都 try-except 包一下。」
- **深挖要点**：（1）**后端/基础设施错误**：网络、超时、API 5xx，通常**重试**或**快速失败**并写入 state.error，通知前端；（2）**Agent 层错误**：工具参数错、解析失败、模型输出格式不对，可能**重试同一请求**、**降级**（如跳过工具继续生成）、或**人工兜底**（如搜索条件人工审核）；（3）**语义/质量类**：幻觉、答非所问、检索无关，难以用异常捕获，需要**评估与人工抽检**；（4）能举例：你项目里「搜索 0 篇」算业务错误写 search_node_error、reading 里 LLM 返回非 JSON 用解析+校验+跳过单篇、RateLimit 用 tenacity 重试。
- **与项目相关**：**全流程**。search_node 写 search_node_error 并走 condition 到 handle_error_node；reading 里单篇解析失败不中断整批、只跳过该篇并打日志；parallel_writing 单节失败上报 error 但 gather(return_exceptions=True) 不拖垮其他节。要能归纳出「分层错误」策略。

---

**问题**：有没有做「检测 + 重试 + 降级或人工兜底」的完整链路？能举一个具体例子吗？

- **浅层回答**：「有重试，tenacity 配了一下。」
- **深挖要点**：（1）**检测**：谁检测（工具返回值、解析 Pydantic、HTTP 状态码）；（2）**重试**：对谁重试（单次 LLM 调用、单篇阅读、整节点）、次数与退避；（3）**降级**：例如检索失败则不用 RAG 直接写、某篇解析失败则丢弃该篇继续；（4）**人工兜底**：搜索条件人工审核、报告生成后是否支持「再编辑再跑」等。能给出**一条完整路径**，例如「阅读单篇 → 解析失败 → 重试 1 次 → 再失败则跳过该篇并记日志 → 其余照常入 KB」。
- **与项目相关**：**阅读 + 写作**。阅读：RateLimit 重试 + 单篇解析失败跳过并记录；写作：RateLimit 重试 + 单节失败上报 SSE、其他节继续。可再补「检索失败时是否降级为无 RAG 写作」的设计。

## 六、评估与迭代维度

**问题**：你怎么评估 Agent / RAG 的效果？有没有指标、测试集或人工抽检？

- **浅层回答**：「跑几个 case 看看效果。」
- **深挖要点**：（1）**指标**：检索有 MRR/Recall@k、NDCG；生成有事实一致性、流畅度、与需求的匹配度（人工或模型打分）；（2）**测试集**：是否有一批固定 query + 期望行为（如必须检索到某类论文、报告必须包含某节）做回归；（3）**人工抽检**：频率、抽哪些环节（检索结果、提取质量、章节是否跑题）；（4）若目前没有，能说出「若要做会先做哪一块」（如先做检索 recall 评估或先做报告章节完整性）。
- **与项目相关**：**全流程**。当前未见自动化评估脚本；可说明「目前依赖人工试用 + 日志排查，若工程化会加：检索 recall 抽样、提取字段与原文一致性抽检、报告结构/关键信息覆盖的 checklist」。

---

**问题**：根据评估结果，你一般怎么迭代？改 prompt、改切片、改模型、还是改流程？

- **浅层回答**：「效果不好就改改 prompt。」
- **深挖要点**：能说清（1）**归因**：先判断问题在检索、在理解、还是在生成，再决定改哪一层；（2）**迭代类型**：prompt（指令、few-shot、输出格式）、切片（chunk_size、按结构）、检索（top_k、阈值、是否 rerank）、模型（换模型、温度）、流程（增加审核节点、增加重试）；（3）**迭代流程**：是否有一版 baseline、改一项做对比、记录配置与结果；（4）项目里是否有配置/版本管理（如 yaml 里不同模型、不同 top_k）便于做对比实验。
- **与项目相关**：**分析 + 写作 + 检索**。prompts、models.yaml、system_params.yaml 等都可作为「可配置项」举例；可说明「若加评估会先固化 baseline 再单变量调参」。

## 七、工程化维度

**问题**：并发和限流你是怎么做的？同时请求多少篇论文、多少节写作？会不会把 API 打爆？

- **浅层回答**：「用 asyncio.gather 并行。」
- **深挖要点**：（1）**并发度**：reading 多少篇同时调 LLM、writing 多少节同时跑；（2）**限流手段**：Semaphore（如 2）、全局限流器、或 API 侧 rate limit；（3）**背压**：若下游慢，是否会堆积任务、内存是否可控；（4）**配置化**：并发度是否可配（如从 config 读），便于不同环境调优。
- **与项目相关**：**阅读 + 写作**。reading_node 用 `Semaphore(2)` + `read_with_limit`；parallel_writing_node 用 `Semaphore(2)` + `write_with_limit`；需能说清「为何选 2」以及「若 API 限流更严会怎么调」。

---

**问题**：有没有对 LLM 请求或检索结果做缓存？缓存的 key 是什么、失效策略呢？

- **浅层回答**：「没有缓存，每次都现调。」
- **深挖要点**：（1）**可缓存对象**：相同 prompt 的 LLM 响应、相同 query 的检索结果；（2）**key 设计**：prompt 的 hash、query+db_id+top_k 等；（3）**失效**：时间 TTL、或知识库更新时失效；（4）若未做，能说明「在什么阶段会考虑加」（如报告生成同一 query 重复跑、或批量评估时）。
- **与项目相关**：**全流程**。当前未实现 LLM/检索缓存；可答「同一 user_request 重复跑会重复调 API，若做会先对 search/reading 的输入做 hash 缓存」。

- **浅层回答**：「看控制台报错。」
- **深挖要点**：（1）**结构化日志**：请求 id、节点名、步骤、关键输入输出（如 query、top_k 结果 id）、耗时；（2）**链路**：是否有一个 trace_id 贯穿整次 run，便于从报告反查到 search/reading/analyse/writing 的每一步；（3）**敏感信息**：日志里是否脱敏（如 API key、用户原文）；（4）**可观测**：是否有进度/状态推送到前端（如你的 BackToFrontData + state_queue），便于复现「卡在哪一步」。

**问题**：多模型切换（例如搜索用 A、阅读用 B、写作用 C）是怎么配置的？改模型需要改代码还是只改配置？

## 八、补充：Human-in-the-Loop 与产品化

**问题**：人工审核（例如搜索条件）是怎么接到流程里的？如果用户一直不点「通过」，流程会一直卡住吗？有没有超时或取消？

- **浅层回答**：「等用户输入，通过 Future 实现的。」
- **深挖要点**：（1）**接入方式**：WebUserProxyAgent.on_messages 里 await waiting_future，前端调 set_user_input 完成 Future；（2）**卡住**：会一直卡住直到用户操作或连接断开；（3）**超时**：是否有 asyncio.wait_for 或 CancellationToken 超时，超时后是走默认查询还是报错；（4）**取消**：用户取消任务时是否能 reject Future 或取消 ainvoke，并清理资源。
- **与项目相关**：**搜索 + 编排**。userproxy_agent 用 Future 阻塞；要能说清「若产品化会加超时与取消，避免长时间挂起」。

**问题**：若要把「报告生成」做成可复现、可回放（例如同一 query 再跑一次得到相同报告），你会保留哪些状态、怎么设计？

- **浅层回答**：「把 state 存下来再跑一遍。」
- **深挖要点**：（1）**确定性来源**：同一 user_request、同一 search_query（含人工审核结果）、同一 search_results，后续 reading/analyse/writing 若模型和温度固定，可近似复现；（2）**要存什么**：至少 user_request、search_query、search_results、extracted_data、analyse_results、sections、writted_sections；或整份 PaperAgentState 序列化；（3）**回放**：从某节点「注入已存 state」再 ainvoke 后续节点，或全图重放；（4）**随机性**：LLM 非确定性、asyncio.gather 顺序等，要说明哪些环节会破坏完全一致。
- **与项目相关**：**编排 + 状态**。State 已包含全链路数据，可说明「复现时从 search_results 或 extracted_data 注入即可跳过前面步骤，便于调试与评估」。
